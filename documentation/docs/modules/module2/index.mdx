:::tip Designing SQL schema for vectorization
When designing a SQL schema for RAG and semantic search, it's crucial to plan ahead for vectorization. Search indexes typically generate embeddings from a single column, meaning that if your relevant data is scattered across multiple columns (e.g., title, summary, tags, content), the search index would need to aggregate them before vectorization. 

**Best practice**: Instead of handling this aggregation at the search index level, merge the data you want to use for semantic search directly in your SQL schema (or even directly in the CSV if you use that format). This can be done by:
- Creating a computed column that concatenates relevant fields
- Adding a dedicated column populated via triggers or stored procedures
- Pre-processing data during insertion to combine searchable content

This approach simplifies your search index configuration, improves performance by reducing processing overhead, and gives you better control over how content is structured for vectorization. For example, combining a document's title, summary, and key metadata into a single `searchable_content` column ensures the embedding model captures the complete context in one operation.
:::

    :::tip
    - **Vectorizer**: Used **at query time** to convert the input query into a vector representation that can be compared against indexed vectors using the algorithm defined in the vector profile. **Critical**: The vectorizer must use the same embedding model as the one used to generate the indexed vectors. Using different models will result in incompatible vector spaces, leading to irrelevant search results (like comparing apples to oranges).
    - **Dimensions**: Specifies the number of vector dimensions returned by the AI Foundry model `text-embedding-3-large`. Note that this doesn't need to match the model's maximum capacity (3072 dimensions). [Reducing dimensions often produces comparable results while decreasing storage requirements and improving cosine similarity calculation performance](https://devblogs.microsoft.com/azure-sql/embedding-models-and-dimensions-optimizing-the-performance-resource-usage-ratio/).   
    - **Include in storage**: While enabled here for demonstration purposes, this option is typically unnecessary in production as it increases index storage size. Only enable it if you need direct access to the raw vector values.
    - **Algorithm**: exhaustiveKnn (K-Nearest Neighbors) provides higher precision but requires more computational resources compared to HNSW (Hierarchical Navigable Small World). Select the algorithm based on your performance requirements and accuracy needs. More details [here](https://www.myscale.com/blog/hnsw-vs-knn-algorithm-comparison/).
    - **Fields**: Making the fields `title` and `link` **retrievable** will include them in the search response and allow the agent to provide correct citations in the Foundry chat playground. Otherwise, you'll get `doc_1`, `doc_2`, etc. as source references.
    :::

4. Finalize the configuration of the skillset by setting the index projections:

    Index projections define which fields from the source documents are included in the indexed chunks. This is crucial for ensuring that the agent has access to all necessary context when retrieving information. By projecting relevant metadata (like author, title, link) alongside the chunked text, the agent can provide more informative and context-aware responses. The mechanism in Azure AI Search is explained [here](https://learn.microsoft.com/en-us/azure/search/search-how-to-define-index-projections?tabs=rest-create-index%2Crest-create-index-projection). 

    ```json title="Skillset index projections"
    "indexProjections": {
        "selectors": [
            {
                "targetIndexName": "ai-knowledge-index",
                "parentKeyFieldName": "parent_id",
                "sourceContext": "/document/pages/*",
                "mappings": [
                    {
                        "name": "text_vector",
                        "source": "/document/pages/*/text_vector",
                        "inputs": []
                    },
                    {
                        "name": "chunk",
                        "source": "/document/pages/*",
                        "inputs": []
                    },
                    {
                        "name": "author",
                        "source": "/document/author",
                        "inputs": []
                    },
                    {
                        "name": "link",
                        "source": "/document/link",
                        "inputs": []
                    },
                    {
                        "name": "title",
                        "source": "/document/title",
                        "inputs": []
                    }
                ]
            }
        ],
        "parameters": {
            "projectionMode": "skipIndexingParentDocuments"
        }
    }
    ``` 

5. Remove the `cognitiveServices`, `knowledgeStore`, and `encryptionKey` properties and save the skillset.

Now that we've created the skillset, the last part of the indexing process is to create the indexer, which ties all other search pieces together to perform the crawl operation.

#### Create indexer

1. Click on _Search management_→_Indexers_ and add new indexer **as JSON** with the following JSON:

    ```json title="Create Indexer JSON"
    {
        "name": "ai-knowledge-indexer",
        "dataSourceName": "ai-knowledge-sql",
        "targetIndexName": "ai-knowledge-index",
        "skillsetName": "ai-knowledge-skillset",
        "fieldMappings": [
            {
            "sourceFieldName": "id",
            "targetFieldName": "chunk_id"
            }
        ]
    }
    ```

    We use a custom field mapping to tell the indexer how to map the source field `id` (from the data source) to the target field `chunk_id` for each processed item (field mappings can't be set via the UI).
    
2. Click on _Save_ to create the indexer. The crawl should start automatically. You can monitor the progress from the indexer page. If your configuration is done correctly, the crawl should be successful with 337 documents indexed:

    :::info
    Comparing the number of processed documents in the indexer (**337**) with the number of items in the index (**1712**) reveals that only the chunks are actually inserted into the index. The original documents are not present in the search index—only their chunked representations with associated metadata. The number of documents in the index will depend on the chunk size.
