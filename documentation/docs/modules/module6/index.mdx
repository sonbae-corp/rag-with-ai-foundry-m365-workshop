### Provision the web application and bot instance

For now, we rely on a local server for the bot logic and a dev bot service instance migrated from the bot framework developer portal. For production scenarios, we need to provision a web application to host the bot logic and a bot service instance configured for production.

1. Add the two following variables in the `variables.local.ps1` file:

    ```powershell
    $ENV_AZURE_SEARCH_APP_OAUTH_CONNECTION_CLIENT_ID="<your-client-id>"
    $ENV_AZURE_SEARCH_APP_OAUTH_CONNECTION_CLIENT_SECRET=<secret>
    ```

    :::info
    Secret for `ENV_AZURE_SEARCH_APP_OAUTH_CONNECTION_CLIENT_SECRET` should be retrieved from your secure secret manager. This is the same client secret used for deployment.
    :::

    These variables correspond to the OAuth2 client ID and secret of the Entra application we created in module 5 to allow the bot service to connect to Microsoft Graph. They will be used in the bot service instance configuration.

1. Run the following command to deploy new Azure resources:

    ```powershell title="Deploy resources"
    .\deploy-infra.ps1 -Module 'Module6' -ConfigureWebApp
    ``` 
    The script will provision three new resources:
    - A **key vault** to store sensitive information used in the web application.
    - A **web application** to host the bot logic.
    - A **bot service** instance configured with MS Teams and Copilot channels and web application as messaging endpoint.

    RBAC permissions are also automatically configured to allow the web application to access the key vault and bot service with its user assigned managed identity.

    At the end of the deployment you should see these three new resources in your resource group:

1. In the `agent.ts`, update the `getCredential()` method to add a condition to use the managed identity when running in Azure:

    ```typescript
    public getCredential(): TokenCredential {

        const isAzure = !!process.env["WEBSITE_SITE_NAME"];

        if (isAzure) {
            console.log("Running in Azure. Using user assigned managed identity credential.");
            
            const clientId = process.env["ENV_AZURE_DEPLOY_USER_MANAGED_IDENTITY_CLIENT_ID"];
            if (!clientId) {
            throw new Error("ENV_AZURE_DEPLOY_USER_MANAGED_IDENTITY_CLIENT_ID is not set");
            }

            return new DefaultAzureCredential({ managedIdentityClientId: clientId });
        } else {
        console.log("Running locally. Using SPN identity credential.");

        return new ClientSecretCredential(
            config.tenantId!,
            config.clientId!,
            config.clientSecret!
        );
    }
    ```
    As explained in [module 5](../module5/#setup-authentication-prerequisites), `DefaultAzureCredential` will automatically use the managed identity assigned to the web application when running in Azure based on the `clientId` and `tenantId` environment variables. We use the variable `WEBSITE_SITE_NAME` to detect if the application is running in Azure App Service as it is only set here.

    :::important Managed identity VS client id/secret
    As a best practice, always use a managed identity to authenticate your applications when running in Azure. This approach eliminates the need to store sensitive credentials like client secrets in your application code or configuration files, reducing the risk of credential exposure. Additionally, with a managed identity, you don't have to worry about rotating secrets or managing their lifecycle, as Azure handles this for you.
    :::

    #### Optimize application files for deployment (Node.js only)

    Azure App Service deployment can be done by simply uploading a [ZIP package](https://learn.microsoft.com/en-us/azure/app-service/deploy-zip?tabs=cli) containing the files for your application. However, for Node.js applications, deployment into Azure App Service expects the entire `node_modules` folder to be uploaded on the server for the runtime code to resolve dependencies. This folder can be quite large and may contain (many) unnecessary files that are not required for the application to run. Including it every time **regardless of the update size** can lead to significant deployment delays:

    <div className="flex justify-center mb-8">
        <img className="w-[30%]" src={WebpackMeme}/>
    </div>

    To avoid this issue, we use [Webpack](https://webpack.js.org/) to reduce the size of JavaScript application deployment packages by bundling only the necessary code and dependencies, significantly speeding up the deployment process. It produces static "standalone" files that already include the specific parts of their dependencies they need to run—and **only those parts**. This process is called **tree-shaking** in JavaScript. After bundled, the application doesn't need the `node_modules` folder anymore.

    1. Add the following development dependencies to the `package.json` and run `npm i`:

            ```json
            "devDependencies": {
                ...,
                "ts-node": "^10.9.2",
                "webpack": "5.97.1",
                "webpack-cli": "^5.1.4",
                "copy-webpack-plugin": "12.0.2",
                "ts-loader": "9.4.4"
            }
            ```

    1. From the root folder of your project, create a new file named `webpack.app.js` with the following content:

            ```javascript
                const path = require("path");
                const CopyWebpackPlugin = require("copy-webpack-plugin");

                module.exports = {
                    mode: "production",
                    devtool: "source-map",
                    optimization: {
                        minimize: false
                    },
                    entry: path.resolve(__dirname, "./src/index.ts"),
                    target: "node",
                    externalsPresets: { node: true }, // in order to ignore built-in modules like path, fs, etc.
                    module: {
                        rules: [
                            {
                                test: /\.ts?$/,
                                use: [                           
                                {
                                    loader: "ts-loader"
                                }          
                                ],
                                exclude: [
                                    /node_modules/
                                ]
                            }
                        ]
                    },
                    resolve: {
                        extensions: [".ts",".js"]
                    },
                    output: {
                        filename: "index.js",
                        path: path.resolve(__dirname, "./dist"),
                        clean: true
                    },
                    plugins: [
                        new CopyWebpackPlugin(
                            {
                            patterns: [
                                { from: path.resolve(__dirname, "./web.config") },        
                                { from: path.resolve(__dirname, "./package.json") }
                            ]
                            }
                        ),
                    ],
                    ignoreWarnings: [
                        {
                            message: /Critical dependency: the request of a dependency is an expression/
                        },
                        {
                            message: /Critical dependency: require function is used in a way in which dependencies cannot be statically extracted/
                        }
                    ]
                };
                ```


        :::tip
        Webpack can be used to bundle various types of JavaScript applications, including web applications, server-side applications, and libraries. In this case, we are using it to bundle a Node.js server-side application for deployment to Azure App Service.
        :::

    1. Update the `package.json` file to add a new build script:

            ```json
            "scripts": {
                ...,
                "start": "node ./index.js",
                "bundle": "webpack --config ./webpack.app.js"
            }
            ```

    1. From the root folder of your project, bundle the application with Webpack running the command `npm run bundle`:

        <div className="flex justify-center mb-8">
            <img className="w-[60%]" src={WebpackOutput}/>
        </div>

        This will bundle all the necessary files into the `dist` folder:

        <div className="flex justify-center mb-8">
            <img className="w-[30%]" src={WebpackDistFolder}/>
        </div>

        :::info
        Notice we copy the `web.config` and `package.json` files to the `dist` folder as they are required by Azure App Service to run the Node.js application (the server will automatically run the `npm run start` command).
        :::

        Now we have the application bundled and optimized for deployment, it is time to deploy it to Azure App Service.

### Deploy to Azure App Service

    1. Open a new terminal from Visual Studio Code and run the following script from the workshop root folder. Use the name of your web application (i.e `app-ai-<id>`) for the `-WebAppName` parameter and the relative of your dist folder (ex: `./apps/typescript/AIWorkshopAgentDevFranck/dist` for Node.js and `./apps/typescript/AIWorkshopAgentDevFranck` for Python) for the `-DistFolderPath` parameter:

        ```powershell title="Deploy to Azure App Service"
        ./deploy-app.ps1 -WebAppName "app-ai-<id>" -DistFolderPath "<your_path>"
        ```

    :::tip Azure CLI over Azure PowerShell
    For app service deployment you can use either Azure PowerShell or Azure CLI. The latter is often preferred for its simplicity and reliability, especially for tasks like deploying web applications.

### Build the Teams package

A Teams package is a ZIP file containing a manifest (`manifest.json`) describing the capabilities of the app and the related resources (icons, localization files, etc.). In our case, it defines the bot ID to use to handle messages for both Copilot (`copilotAgents`) and Teams (`bots`):

        ```json
        {
            "$schema": "https://developer.microsoft.com/en-us/json-schemas/teams/v1.24/MicrosoftTeams.schema.json",
            "manifestVersion": "1.24",
            ...
            "copilotAgents": {
                "customEngineAgents": [
                    {
                        "type": "bot",
                        "id": "${{BOT_ID}}"
                    }
                ]
            },
            "bots": [
                {
                    "botId": "${{BOT_ID}}",
                    "scopes": [
                        "copilot",
                        "personal",
                        "team"
                    ],
                    "supportsFiles": false,
                    "isNotificationOnly": false,
                    "commandLists": [
                        {
                            "scopes": [
                                "copilot",
                                "personal"
                            ],
                            "commands": [
                                {
                                    "title": "How can you help me?",
                                    "description": "How can you help me?"
                                }
                            ]
                        }
                    ]
                }
            ],
            ...
        }
        ```

This manifest is based on a JSON schema](https://learn.microsoft.com/en-us/microsoft-365/extensibility/schema/?view=m365-app-1.24), regularly updated by Microsoft according to new available capabilities. The package is then uploaded to the Teams admin center to deploy the application to your tenant and make it accessible to users. To generate it, you have multiple options:

- Manually by creating a ZIP archive (not recommended)
- Using the [Microsoft 365 Agents Toolkit CLI](https://learn.microsoft.com/en-us/microsoftteams/platform/toolkit/microsoft-365-agents-toolkit-cli) (`atk`)
- Using the [Microsoft 365 Agents Toolkit VS Code extension](https://marketplace.visualstudio.com/items?itemName=TeamsDevApp.ms-365-agents-toolkit)

In our case, we will use the CLI to build the package. The Microsoft Agents Toolkit (extension or CLI) works with the concept of environment defined in the `env` folder. Each environment can have its own configuration and settings (like different bot IDs, endpoints, etc.). By default, the toolkit creates the following environments:
- `playground`: used for testing in the Microsoft 365 Agents Playground (without any connectivity to Azure resources or Microsoft 365 services)
- `local`: used for local debugging, when the app is running on your local machine.
- `dev`: used for remote debugging and testing, when an app is already deployed in Azure.

:::important Network access for the bot service
For the Teams client to be able to call your bot in Azure, this one must have the **public access** enabled:

        <div className="flex justify-center mb-8">
            <img className="w-[40%]" src={BotPublicAccess}/>
        </div>
:::

You can also create your own custom environments (like `prod`, `staging`, etc.) by creating new `.env` files in the `env` folder and adding the related configuration.

1. From the root folder of your project, update the `.env.dev` file with the following variables (leave the other variables untouched as they are not used):

        ```env
        BOT_ID=<ex: 25934a87-2edc-4d29-a9d1-b185fcbfaf47>
        TEAMS_APP_ID=<ex: 95eff6fd-ed69-4604-8cab-db337ed13eb0>
        ```
    Where:

    - `BOT_ID`: The bot id in Azure:

        <div className="flex justify-center mb-8">
            <img className="w-[40%]" src={BotId}/>
        </div>

    - `TEAMS_APP_ID`: Unique ID of the application. For the first deployment, you can generate a [new one](https://www.guidgenerator.com/). For subsequent deployments, use the same ID to update the existing application in the Teams admin center. 

1. From the root folder of your project, in a new terminal, run the following command to build the Teams package for the `dev` environment:

        ```powershell title="Build Teams package"
        atk package --env dev
        ```

    This will generate a ZIP package in the `packages` folder containing the `manifest.json` and related resources with the correct bot ID and endpoints for the `dev` environment:

    <div className="flex justify-center mb-8">
        <img className="w-[100%]" src={AtkOutput}/>
    </div>
 

    :::tip
    You can also use the Agents 365 SDK VS Code extension to package an application:
    <div className="flex justify-center mb-8">

    </div>

    In real project scenarios, this step should be automated as part of a CI/CD pipeline using tools like GitHub Actions or Azure DevOps to ensure consistent and repeatable deployments. Additionally, as a best practice, you should update the version in the `manifest.json` and ZIP package name accordingly. **You can't deploy a package with the same version twice in the Teams admin center**.

## Deploy the package

To deploy a Teams package, you must have the "Teams administrator" role. In organizations where this role is restricted, coordinate with your Microsoft 365 admin team to complete this step.

1. From the [Azure portal](https://portal.azure.com/?feature.msaljs=true#view/Microsoft_Azure_PIMCommon/ActivationMenuBlade/~/aadmigratedroles/provider/aadroles), activate your Teams Administrator role.

1. Go to the [Teams administration center](https://admin.teams.microsoft.com/) (you may have to logout/login again) and go to _Teams apps_→_Manage apps_:

    <div className="flex justify-center mb-8">
        <img className="w-[60%]" src={TeamsManageApps}/>
    </div>

1. On the top right corner, select _Actions_→_Upload a new app_ and upload the same ZIP package created previously:

    <div className="flex justify-center mb-8 space-x-2">
        <img className="h-[130px] w-[295px]" src={TeamsUploadNewApp}/>
        <img className="w-[50%]" src={TeamsAppAdded}/>
    </div>

    :::important 
    - In Teams, you can restrict an application to specific user groups by configuring app permission policies in the Teams admin center. This is particularly useful in production scenarios to control which users or groups have access to specific apps within the organization, for instance in a UAT phase when only specific business people need to access.
    - The application can take time to appear in the teams app catalog (from minutes to hours) depending on Microsoft 365 tenant configuration.
    - Making an update to your app (ex: in the web application) doesn't require you to redeploy a package. You only need to redeploy the package when you make changes to the app manifest (title, description, capabilities, etc.).
    :::
